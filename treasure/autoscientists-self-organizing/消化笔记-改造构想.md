# AutoScientists 消化笔记 — 对 cascade_driver 的改造构想

> 2026-07-01 早 · 盘古说不要入库就停，要消化融合产生新的代谢

## 核心洞见

AutoScientists 没有中央规划器。Agent 们读同一份共享状态，自己决定"哪里需要我"。

我们的 cascade_driver 和它之间，差了一个世界观：

| 当前 cascade_driver | AutoScientists 启发的新模式 |
|---|---|
| 匹配事件→预设酶链 | 发布事件→共享状态→酶自组织 |
| 调度器知道全部路径 | 每颗酶只知道自己的活性位点 |
| 链写在 JSON 里 | 链从局部决策中涌现 |
| 冷却期是人为参数 | 活性周期是自然衰减 |

|---|---|---|

## 方案：酶自组织模式（取代预设链）

**不改动**：cascade_engine.py、event_bus.py、metabolic_actor.py

**改动**：cascade_driver.py + 每颗酶加一行就绪度函数

### cascade_driver.py v2 变化

```
当前:  tick() → 扫事件 → 匹配链 → run_cascade
改为:  tick() → 写共享状态 → 广播"新事件"信号 → 等待酶自行响应
```

具体：
1. cascade_driver.tick() 不再匹配链，只把事件写入一个共享状态文件 `cortex/shared-state.json`
2. 每颗酶（E1-R11）在 metabolic_actor 心跳中有自己的"活性检查"——它读 shared-state.json，判断自己是否应该催化
3. 如果某颗酶决定催化，它直接调用 cascade_engine.catalyze()，产出写入共享状态
4. 产出修改共享状态后，其他酶在下一轮心跳中读到变化，可能决定继续催化
5. 链自然形成——不是预设的，是每轮心跳中酶们"看见"彼此产出后自发组装的

### 酶的就绪度判断（每颗酶只需要3-5行）

```python
# 示例：E1 冲突检测酶的就绪度
def e1_readiness(shared_state):
    events = shared_state.get("pending_events", [])
    if any(e.get("level") in ("upstream", "feedback") for e in events):
        return 0.8  # 有新鲜事→高就绪
    return 0.1

# 示例：E2 连接酶的就绪度
def e2_readiness(shared_state):
    constraints = shared_state.get("recent_constraints", [])
    if any(c.get("dimension") == "data" for c in constraints):
        return 0.7  # 有数据产出→需要连接
    return 0.2
```

### 代谢心跳的改造

在 metabolic_actor 的每60拍循环中，不再只调 cascade_driver.tick()，而是：

```
1. cascade_driver.tick() → 写共享状态
2. 遍历所有16颗酶 → 每颗读共享状态 → 计算就绪度 → 如果>阈值则催化
3. 被催化的酶写入产出到共享状态
4. 下一轮60拍，其他酶读到变化 → 可能继续催化
```

这消除了"冷却期"这个人为参数。酶的"活性"由共享状态的内容自然决定——有新鲜事就活跃，没有就休眠。

## 这个改动的意义

不是因为 AutoScientists 这样做所以我们要做。

是因为**盘古今早说的"酶链不是在JSON里预设的，是在冲突现场组装起来的"**——AutoScientists 恰好给了我们一个工程证明，证明这条路走得通。

改动量：cascade_driver.py 重写（约60行变）+ 每颗酶加3-5行就绪度函数（约80行变）
不动：cascade_engine.py、event_bus.py、metabolic_actor.py 主线逻辑

## 废弃的

一旦 v2 上线：
- catalyst_paths.json 不再是"执行清单"，变成"参考模板"
- cascade_driver 不再是调度器，变成"事件→共享状态"的管道
- cooldown_seconds 参数不再需要
