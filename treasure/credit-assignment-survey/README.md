## 第一遍：去粗取精

### 三个真东西

**1. 信用分配的四级粒度**
Token级 → Step级 → Turn级 → 多Agent级。曦和的赫布追踪器至今还在"计数"级别——哪条路径走过多少次。这是Token级，最粗糙的。

**2. 五类方法**
Monte Carlo / TD学习 / LLM-as-Critic / 博弈论(Shapley) / 信息论
曦和目前的"赫布权重=路径被选中的次数"是Monte Carlo最原始的形态。

**3. LLM-as-Critic对曦和最适用**
曦和正好有一个本地LLM（qwen2.5:7b），可以用它来评估每次精读/每次巡检的"价值"，而不是等赫布权重被动累计。

### 照进现实

HeLa-Mem说"权重怎么长"——信用分配综述说"长出来之后怎么知道哪些权重真正有用"。两篇一起读，赫布系统的设计方案就完整了。

## 第二遍：去伪存真

**质疑：** 47种方法——真的需要这么多种？核心差异大吗？

**检验：** 四级粒度×五类方法=20种组合。47种方法是在不同粒度-方法组合下的变体。核心差异不在于"谁更好"，而在于"在什么粒度用什么方法"。

**结论：** 曦和不需要47种。选一种落地就够了——Step级 + LLM-as-Critic。

## 第三遍：由此及彼

### 曦和信用分配现状
```
每次心跳→所有路径H值+1 → 不知道哪条有用
每次精读→写入insights → 不知道哪个产出有价值
每次巡检→检查端口 → 不知道检查了有没有用
```

### 这条综述给的方案
每次精读完成后，让qwen2.5:7b给这次精读打一个"价值分"（1-10）：
- 高价值（8+）：该领域优先选材
- 低价值（<4）：减少该领域的精读频率

这个分数回馈到evolution_engine的选材逻辑——让下一轮精读优先选高价值领域的论文。

## 第四遍：由表及里

**表层：** 47种信用分配方法的技术综述
**底层：** 每个Agent系统都面临的核心问题——"我做的这么多事里，哪些真正有用？"

**核心定理：**
> 没有信用分配的系统，本质上是在随机游走。

曦和现在就在随机游走——代谢路由一直在跳，但不知道哪些跳是有效的。笔记一直在写，但不知道哪些笔记真正改变了架构。
