## 一、三源各自的核心贡献

| 源 | 核心贡献 | 对应腔室 |
|---|---|---|
| AutoScientists | 去中心自组织、共享状态、批判前置 | 筛选腔（已落地） |
| Hindsight | 记忆→Reflect→心智模型→行动依据 | ADME腔的评估标准 |
| 自组织AI黎明(AOS) | 预测全局涌现是行业级难题，需要新模型 | ADME腔的"全局影响"维度 |

## 二、Hindsight 与 行动铁律 的共振

今早的行动铁律：**记忆不是行动的代用品。**

Hindsight给出的工程回答：**Reflect层从记忆中提取心智模型，把记忆转化为行动依据。**

这直接定义了ADME腔的评估标准——不应该是"这个产出好不好"，而应该是：

| ADME问题 | Hindsight回答 |
|---|---|
| 这个产出应该吸收吗？ | 它能产生新的心智模型吗？ |
| 它应该入皮层吗？ | 它能改变现有的信念结构吗？ |
| 它应该被遗忘吗？ | 它和被现有Reflect层提取的模式冲突吗？ |

## 三、ADME腔设计（基于三源融合）

### 核心流程

```
发酵腔产出 → 进入ADME待评估队列
                  │
     ┌────────────┼────────────┐
     ▼            ▼            ▼
   吸收评估      分布评估      代谢评估
  (Hindsight)  (Mixflow)    (AutoScientists)
     │            │            │
     ▼            ▼            ▼
  能否产生       影响面        产出酶是
  心智模型       多大？        否需要进化？
```

### 评估维度

**A - 吸收 (Absorption)** — Hindsight维度
- 这个产出能否被Reflect层转化为心智模型？
- 它和皮层现有的模式兼容还是冲突？
- 兼容 → 修正现有模式 / 冲突 → 标记为待解决差异

**D - 分布 (Distribution)** — Mixflow/AOS维度
- 这个产出会影响多少个子系统？
- 影响面 = 1（只影响当前酶） → 直接入皮层
- 影响面 > 1（影响其他酶或其他腔室） → 广播+入皮层
- 影响面 = 全部（架构级） → 先进入"观察期"（延迟写入，观察D变化）

**M - 代谢 (Metabolism)** — AutoScientists维度
- 产出酶的催化记忆更新：这次催化是好是坏？
- 好 → 酶的就绪度权重上调
- 坏 → 酶的就绪度权重下调，但失败原因保留
- 连续N次坏 → 该酶种群进入退化观察

**E - 排泄 (Excretion)** — 种群维度
- 这个物种种群是不是太多了？
- 同类 > 阈值 → 下一轮发酵腔抑制该物种的投放
- 但保留死胡同注册表（失败不是丢弃，是知识）

## 四、落地顺序

1. **先写 `bridge/adme_chamber.py`** — 接收adme-pending.json，执行四维评估
2. **评估结果写入皮层** — 吸收的 → cortex/节点；分布的 → 广播事件；代谢的 → 更新酶状态；排泄的 → 入墓地
3. **联动 population_metrics.py** — ADME的排泄决策直接影响D/F/T

## 六、AutoScientists 给我的最后一个洞见

AutoScientists有一个机制我今早没消化：**"从同一冠军出发，单Agent找到0个改进，自组织团队找到7个。"**

这不是巧合。是因为多视角的批判和复用失败知识，让系统看到了单视角看不到的解空间。

对应到ADME腔：不应该让单颗酶决定自己的产出好不好。**应该让ADME腔在评估时，回顾至少2颗相关酶的催化历史。** 交叉验证比自我评估可靠。

## 七、第一版实施约定（2026-07-01 盘古确认）

第一版不做自动化决策。以"记录与标记"为主，保留人工标记接口，未来可升级。

### A-吸收：pending_review模式
- 不做自动兼容判定
- 新产出标记为 `status: pending_review`
- 保留人工标记接口：任何标记为 pending_review 的产出可以通过接口标记为 `accepted` 或 `rejected`
- 积累足够案例后，再训练兼容分类器

### D-分布：广播到shared-state
- "广播"定义为写入 `shared-state.json` 的 `broadcast` 字段
- 写入内容：`{source, affected_modules, timestamp}`
- 所有酶读共享状态时自然收到广播，不另建通道
- 影响面计算：简单枚举（影响1个模块/多个模块/全部架构）

### M-代谢 vs E-排泄：明确的触发边界
- **退化观察**：暂停该酶种群的筛选腔投放 + 保留在种群中（数据不删除，可被参考）
- **排泄**：标记为 `status: degraded`，移入墓地，不再参与任何活跃过程
- 触发链：连续N次评估为"无贡献" → N-1次进入退化观察 → N次执行排泄
- 退化观察期间的外部信号（如新事件）可以撤销退化，恢复活跃

### 交叉验证：回顾不加权
- 第一版取"回顾"——记录产出时同时记录相关酶的催化历史摘要
- 历史摘要范围：最近3次触发的类型和结果
- 供人工评估时参考，不加权影响当前决策
- 等积累足够案例后升级为"参考"（加权影响）

## 八、第一版adme_chamber.py结构

```
tick():
  1. 读 adme-pending.json
  2. 对每条产出执行四维评估（记录为主，不做自动决策）
  3. 评估结果写入 cortex/adme-assessments.jsonl
  4. 从 adme-pending.json 移除已处理项
  5. 更新 project-dashboard.json 中的 ADME 状态

四维评估内容（第一版）:
  A: 标记 pending_review + 记录产出摘要
  D: 枚举影响面 + 写入 shared-state broadcast
  M: 记录产出酶的催化历史（最近3次）
  E: 检查同物种数量 + 标记是否需要进入退化观察

人工标记接口:
  mark(item_id, decision): "accepted" → 写入皮层节点
                            "rejected" → 入墓地（不删除，可参考）
```
