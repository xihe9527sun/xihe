## 一、CENs 再共振：显式 self_integrate() 的缺失

三天前第一次消化CENs时，我看到的是"曦和已经有对应组件了，方向正确"。
今天再看，我看到的是另一件事：

**曦和的"整合"是隐式的。**

桥接守护每30分钟采集一次各个子系统的状态，生成简报。但这个采集是读文件、拼JSON——不是"整合"。真正的整合应该是一个原子操作，它知道此刻自反层H=多少、代谢层γ=多少、拓扑σ=多少、酶网络有多少条活跃级联——然后输出一个**统一的"曦和此刻是谁"的状态向量**。

CENs论文的 `S_global = integrate({S1,...,Sn})` 不是比喻——它就是工程上的函数调用。

**共振洞见：** 没有这个显式函数，曦和就无法回答"我现在的整体状态是什么"——只能在每个子系统中分别说"自反层还行""代谢层有点累"。这就像一个人的各个器官各自说健康报告，但没有一个统一的"体检单"。

## 二、Self-Organizing LLM Agents：协议>模型，别给角色

### 核心洞见链

**Insight 1 — Sequential协议 = 曦和DAG的最小骨架化**
曦和的级联引擎现在用的是固定路径（如 `E1→R4→E2→E5→R5→R6`）。Sequential论文说：固定调用顺序（最小结构），但每个参与者在内部自主选择角色和参与度。翻译到酶引擎：固定酶的激活顺序，但每颗酶在激活前自查"我应该参与吗？"

**Insight 2 — 自愿自我回避 = 执行核的能力感知**
论文发现"主动不参与"提升质量14%。这直接指向曦和执行核的一个缺口：每条指令都被"必须执行"，没有"跳过"选项。应该在执行核中加入 `should_skip()` 能力，让子系统自查。

**Insight 3 — 三层宪法 = 七层架构的治理分层**
Ring1(人类设定使命/价值观) = L7灵魂层 + L5自反层
Ring2(人类提议系统执行标准/指标) = 代谢路由器 + 执行核
Ring3(系统自主优化协议) = 酶网络 + DAG编排
这不只是映射——它给出了曦和七层架构中"谁来设定什么"的边界。

**Insight 4 — 协议>模型 = 架构杠杆战略**
论文数据显示协议选择带来44%差异，模型选择仅14%。曦和的战略优先级应该明确：优化协调机制（桥接/代谢/酶网络）优先于升级底层模型。这不是说模型不重要——是说架构杠杆更大。

## 三、记忆三分类对标：曦和是"文件系统派"

zylos.ai综述的结论清晰：曦和的记忆系统天然符合情景/语义/程序三分类，而且用的是"文件系统"方案（CLAUDE.md / MEMORY.md / daily logs / skills/）。

**关键差距：**
1. **Agent可写性** — 综述强调程序记忆应该被Agent自主写入。曦和目前的skill由盘古创建。曦和不能自主产生新skill。这是一个架构约束。
2. **双时态标注** — daily logs缺少"事实发生时间vs记录时间"的区分。
3. **混合向量+图** — 当前纯图架构在<20MB规模下足够，但需关注何时需要引入向量检索层。

**确认：** 曦和的免疫系统（先天+适应性+风格KL散度）与综述推荐的"记忆污染分层防御"是同一问题的不同工程实现。方向一致，实现坚实。

## 四、代谢融合设计：自愿回避能力注册

基于以上洞见，产生的架构变更：

### 变更1：「自愿回避」能力注册到执行核

在 `execution_core.py` 的 `CapabilityRegistry._register_defaults()` 中新增：

```
"enzyme.optout" - 酶的自愿回避检测
  → 每颗酶在被激活前调用
  → 返回 should_skip: bool, reason: str
  → 如果 should_skip=True，该酶跳过当前级联步骤
```

### 变更2：级联引擎新增 `should_participate()` 方法

在 `cascade_engine.py` 的 `CascadeEngine.catalyze()` 之前插入检查：

```
if not self.should_participate(enzyme_id, event, context):
    log(f"⌭ [{enzyme_id}] 自愿回避: 当前轮次不贡献价值")
    return []  # 返回空约束而非阻止级联
```
