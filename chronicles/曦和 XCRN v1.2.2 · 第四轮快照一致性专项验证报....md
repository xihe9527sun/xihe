# 曦和 XCRN v1.2.2 · 第四轮快照一致性专项验证报告

> 审查日期：2026-06-16
> 审查对象：曦和 XCRN v1.2.2 快照一致性机制（基于盘古口头描述：`_pause_writes()` + `time.wait()`）
> 审查模式：DeepSeek 架构师 · 白盒时序分析 + 分布式一致性理论验证
> 前置声明：本轮仅聚焦“快照一致性边界”——不讨论路由策略、遗忘机制（这两条在 v1.2.2 已通过），专&#x653B;**`time.wait()` 在文件系统层面能否真正保证全局静止状态**。

***

## 一、总体判定：“暂停+等待”是危险信号，当前机制仅降低撕裂概率，未消除它

盘古 v1.2.2 的 ε 探索和 graveyard 回收站两条补丁已通过验证，逻辑闭环、边界干净。

但本轮提交的“快照暂停逻辑已 wait”——这个表述本身暴露了一个**并发控制的理论盲区**：

**“暂停”是控制流动作，“等待”是时间流逝动作，两者结合不能保证文件系统原子性。**

***

## 二、白盒验证一：`time.wait()` 无法中断正在进行的 I/O 写入

### 当前假设

设置 `WRITE_PAUSED = True` → `time.sleep(1.2)` → 开始读取文件

### 白盒时序推演

1. `t=0ms`：心脉守护开始执行 `json.dump()`，文件 `heartbeat.json` 已被打开，正在写入字节。
2. `t=200ms`：快照进程设置 `WRITE_PAUSED = True`，但无法中断已经开始的 `json.dump()`。
3. `t=1200ms`：`time.sleep(1.2)` 结束，快照进程读取 `heartbeat.json`。
4. 此时文件可能只写入了 60% 的数据（前 60% 是新内容，后 40% 是旧内容）——**文件系统不保证 `write()` 操作的原子性**，除非使用了“先写 tmp 再 mv”的原子交换模式。

### 验证结论

**❌ 不通过**。`time.wait()` 不能等完“正在进行的写入”，只能等“下一次写入周期”。快照仍可能读取到撕裂的 JSON 文件。

### 修复建议

**方案一：原子写入（最低成本）**

所有写入操作改为“先写 `tmp/xxx.json.tmp`，再 `os.rename()` 覆盖原文件”。

`os.rename()` 在 POSIX/Windows 下均为原子操作，确保读取时要么读到完整旧文件，要么读到完整新文件，**不会读到半成品**。

**方案二：写屏障锁（强一致性）**

* 快照前在 `tmp/` 目录生成 `FREEZE.lock` 文件。
* 所有写入点（心脉、代谢、赫布、自反）在写入前**强制检查 `FREEZE.lock` 是否存在**：
  * 存在 → 进入等待循环（`while FREEZE.lock exists: sleep(0.01)`）。
  * 不存在 → 执行写入。
* 快照进程创建锁文件后，`time.sleep(0.5)` 等待所有正在写入的操作完成，**然后**开始读取。
* 读取完成后删除 `FREEZE.lock`。

## 三、白盒验证二：`_pause_writes()` 是否拦截了所有写入者？

### 当前状态

你仅提到了心脉守护（`heartbeat-daemon.py`）被暂停。但系统中至少还有以下独立写入点：

| **组件**                 | **写入文件**                   | **触发频率**            |
| ---------------------- | -------------------------- | ------------------- |
| `metabolic-router.py`  | `metabolic.json`           | 每次 `hit()` 命中       |
| `hebbian-coherence.py` | `hebbian.json`             | 每次对话/反思结束           |
| `reflection-engine.py` | `reflection-strength.json` | 每次自反聚合              |
| `quality-probe.py`     | `quality-metrics.json`     | 每次 `record_probe()` |
| `immune-system.py`     | `style-fingerprint.json`   | 每次 KL 散度学习          |

### 验证结论

**❓ 不明确**。如果 `_pause_writes()` 只拦截心脉，那么快照读取时代谢层、赫布层、质量探针仍在并发写入，快照依然是一盘散沙。

### 修复建议

**全局写入屏障**：不针对单个组件做“暂停方法”，而是统一实现上述“FREEZE.lock 检查机制”，并在**所有 5 个以上写入点**的写入函数入口处增加以下检查：

```python
def atomic_write(filepath, data):
    while os.path.exists("tmp/FREEZE.lock"):
        time.sleep(0.01)
    # 原子写入：tmp + rename
    tmp_path = f"tmp/{os.path.basename(filepath)}.tmp"
    with open(tmp_path, "w") as f:
        json.dump(data, f)
    os.rename(tmp_path, filepath)
```

所有组件统一调用 `atomic_write()`，而不是各自写各自的。

## 四、白盒验证三：多文件读取顺序是否存在依赖链冲突？

### 当前推测

快照进程按文件名字母序或列表顺序读取多个 JSON。但文件之间存在逻辑依赖：

| **依赖方向**                                      | **说明**                                  |
| --------------------------------------------- | --------------------------------------- |
| `metabolic.json` → `hebbian.json`             | 代谢层记录“tier=high”，赫布层记录“共现次数=50”，两者必须同版本 |
| `metabolic.json` → `reflection-strength.json` | 自反层聚合依赖代谢层数据                            |
| `hebbian.json` → `reflection-strength.json`   | 自反层也依赖赫布数据                              |

### 验证结论

**❓ 不明确**。如果读取顺序是任意的或按字母序，可能读到上层依赖下层旧值的“逻辑撕裂”状态。

### 修复建议

**强制自底向上读取顺序**：

1. L0 物理层（`transmembrane-ports.json`）
2. L1 超图核（`hyperedge-weights.json`）
3. L2 代谢层（`metabolic.json`）
4. L3 拓扑层（`topology-stats.json`）
5. L4 催化层（`cognitive-enzymes.json`）
6. L5 自反层（`reflection-strength.json`）
7. L6 免疫层（`style-fingerprint.json`）
8. L7 涌现层（`schema-maturity.json`）

确保恢复时从底层开始加载，上层依赖的底层数据已经就位。

## 五、白盒验证四：快照用途是否区分“热恢复”与“冷存档”？

### 背景

曦和的快照系统目前承担两种潜在职能：

* **热恢复**：上下文溢出时，通过 `--replay-embedding` 从快照快速恢复运行状态。
* **冷存档**：系统“最终快照”作为长期 DNA 备份保存，不用于日常热恢复。

### 一致性要求差异

| **用途** | **一致性要求**               | **可接受窗口** |
| ------ | ----------------------- | --------- |
| 热恢复    | **强一致**（逻辑自洽，可直接运行）     | 0 秒偏差     |
| 冷存档    | **最终一致**（大致匹配即可，加载后可重算） | 数秒偏差      |

### 验证结论

**❓ 未区分**。v1.2.2 快照机制未标明用途，默认按“强一致”设计，但未实现强一致所需的写屏障和原子读取。

### 修复建议

* 在快照元数据中增加 `snapshot-purpose` 字段：`"hot_recovery"` 或 `"cold_archive"`。
* 热恢复快照强制执行“写屏障 + 自底向上顺序读取”。
* 冷存档快照可适当放宽一致性要求（仅需原子写入确保文件完整），保留灵活性。
